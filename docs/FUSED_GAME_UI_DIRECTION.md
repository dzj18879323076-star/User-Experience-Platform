# A+C 融合版游戏化方向

## 结论

推荐把 A「地图闯关」作为主导航，把 C「证据调查板」作为过关机制，形成「地图推进 + 证据过关」的新人训练产品。

这个方向比单纯表单更像游戏，也比纯证据板更适合作为可复用的新人工具：新人知道自己在第几关、为什么要做这一关、还差什么证据才能解锁下一关。

## 设计判断

### A 的价值：建立任务动机

地图闯关解决的是新人任务的心理模型问题。用户一打开产品，先看到 6 个关卡和 Boss 关，而不是一组字段。这会让体验报告从「填完材料」变成「完成调查路线」。

适合保留：

- 总进度、当前关、下一关、Boss 关
- 关卡解锁状态
- 每关的主题和胜利条件

不适合单独使用：

- 如果只有地图，关卡仍然可能退化成普通任务清单
- 对产品经理最重要的证据链和推理质量没有被显性训练

### C 的价值：训练产品思考

证据调查板解决的是报告质量问题。它迫使新人把观察拆成用户目标、路径断点、产品归因、待验证证据，而不是只写主观感受。

适合保留：

- 证据卡
- 推理评分
- 待补证据
- 阶段战报

不适合单独使用：

- 视觉上更像专业工作台，游戏推进感不足
- 新人可能不知道证据卡和完整体验路线之间的关系

## 融合后的核心循环

1. 选择当前关卡
2. 接收一个真实用户任务
3. 体验产品路径并收集证据
4. 把证据归类到证据袋
5. 系统按过关条件判断缺口
6. 提交战报并解锁下一关

这套循环的重点不是「好看」，而是让新人每次体验都产生可复用的产品判断。

## 当前原型

入口：

`web/prototype.html?variant=D`

页面结构：

- 左侧：新人生活服务调查地图
- 右侧：当前案件、证据袋、本关过关条件
- 底部：提交战报并解锁下一关

## 对 B 的定位

B「Game Master 控台」暂时不作为主界面。它更适合作为后续能力层：

- 当用户提交战报后，AI Game Master 做追问
- 当证据不足时，AI 给出补证任务
- 当关卡完成后，AI 生成阶段汇报卡

也就是说，B 适合作为交互机制，不适合作为第一版的主视觉结构。

## 下一步产品化建议

优先把 `variant=D` 作为主方向继续迭代：

- 把关卡节点和证据卡接入真实数据结构
- 每关定义 3-5 个硬性过关条件
- 把「证据袋」沉淀为报告结构
- 引入 AI 追问，但只在证据不足时触发
- 保留 A/B/C 作为设计备选和复盘依据
