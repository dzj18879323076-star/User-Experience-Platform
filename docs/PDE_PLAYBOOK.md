# PDE Playbook：用 Codex App 推进新人闯关项目

## 1. 本项目的 PDE 定义

这里的 PDE 不是“写一份文档”，而是把一个业务想法推进成可体验、可验证、可迭代的产品工程项目。

本项目的 PDE 闭环是：

```text
业务命题 -> PRD -> 技术方案 -> Web MVP -> 真实试跑 -> 反馈迭代 -> 部署/复用
```

## 2. 从视频笔记迁移过来的工作法

### 2.1 项目以文件夹为单位管理

Codex App 的项目管理单位是文件夹。因此本项目应保持清晰目录：

```text
life_service_onboarding_quest/
  AGENTS.md
  docs/
  web/
  config/
  runs/
  templates/
```

`AGENTS.md` 是项目级记忆，后续新对话进入该项目时，应先读取它理解项目背景。

### 2.2 复杂任务先计划再执行

当任务涉及产品方向、技术方案、前端实现、部署、AI 接入等复杂改动时，先做计划：

```text
目标是什么？
当前阶段是什么？
要改哪些文件？
验收标准是什么？
是否需要用户确认？
```

对本项目而言，适合先计划的任务包括：

- 改关卡机制
- 改产品信息架构
- 引入后端或数据库
- 接入 AI Game Master
- 接入飞书文档同步
- 部署上线

### 2.3 Steer：发现方向错就及时纠偏

本项目已经出现过一次重要纠偏：

> 不能把闯关训练继续做成飞书文档交互，而应该做成 PDE 项目，用网页承接核心体验。

后续如果发现 Codex 偏向“写文档”而不是“做产品”，应立即纠偏到：

```text
PRD 是否要更新？
技术方案是否要更新？
网页产品是否要改？
如何验证？
```

### 2.4 Git 管理与回滚

项目应尽快进入 Git 管理状态，便于回滚和并行探索。

建议：

- 初始化 `.gitignore`
- 将 `life_service_onboarding_quest/` 纳入版本管理
- 每完成一个可验证阶段就提交
- 大改版使用分支或 worktree

适合分支化的方向：

- `feature/ai-game-master`
- `feature/leader-review`
- `feature/feishu-sync`
- `feature/static-deploy`

### 2.5 Worktree 并行探索

当同时探索不同方案时，使用 worktree 或分支隔离。

例子：

- 一个分支继续做静态 Web MVP。
- 一个分支探索 React/Next.js 版本。
- 一个分支探索飞书应用/妙搭承接。

不要在主线里同时混入多个大方向。

### 2.6 Skills：把稳定工作流封装

如果本项目跑通，可以沉淀成自定义 Skill：

```text
新人闯关项目 Skill
输入：业务主题、岗位、训练目标
输出：关卡设计、PRD、技术方案、Web MVP、试跑计划
```

这个 Skill 可以复用到其他新人训练：

- 商家产品新人
- 交易产品新人
- 内容生态新人
- 评价治理新人

### 2.7 MCP：后续连接外部系统

MCP 不是 MVP 必需项，但适合后续接入：

- 数据库：保存多人训练记录、评分、报告
- 飞书：同步报告、通知 leader、创建任务
- GitHub/代码平台：管理项目版本和部署
- 部署平台：发布静态站点或全栈应用

## 4. 下一阶段建议

### Phase 1：让 Web MVP 承接真实试跑

目标：

- 把关卡 1 的真实观察录入网页。
- 验证表单、保存、汇报卡草稿是否可用。
- 找出网页交互缺口。

验收：

- 关卡 1 能在网页里完整保存。
- 页面刷新后数据不丢。
- 生成的阶段汇报卡能作为 AI 二次加工输入。

### Phase 2：优化关卡 2 体验

目标：

- 支持 3 个样本对象的结构化填写。
- 区分有用评价、无用评价、评价缺口。
- 输出有效/无效评价样本库。

验收：

- 关卡 2 表单不再只是单组字段，而是支持多样本记录。
- 汇报卡能提炼评价质量规则。

### Phase 3：AI Game Master 接入

目标：

- 把规则生成升级为 AI 追问、评分、汇报卡。
- 每个关卡有定制化 prompt。

验收：

- 用户提交观察后，AI 能提出 3 个高质量追问。
- AI 能按评分维度输出分项得分。
- AI 能生成可写入报告的阶段汇报卡。

### Phase 4：leader 评阅模式

目标：

- 增加只读评阅视图。
- 展示新人完成度、观察质量、机会点。

验收：

- leader 能快速判断新人是否完成体验。
- leader 能看到关键问题和待验证问题。

### Phase 5：部署与复用

目标：

- 将 Web MVP 部署到可分享地址。
- 支持后续新人使用。

优先方案：

1. 静态站点部署。
2. 飞书应用/妙搭承接。
3. 全栈应用 + 数据库。

## 5. 当前 Web MVP 缺口

| 缺口 | 影响 | 优先级 |
|---|---|---|
| 关卡 2 不支持多样本结构 | 看评裁判所体验不够贴合任务 | 高 |
| 汇报卡仍是规则模板 | 缺少 AI Game Master 价值 | 高 |
| 没有 leader 视图 | 无法验证团队复用场景 | 中 |
| 不能同步飞书 | 报告输出仍需复制粘贴 | 中 |
| localStorage 仅本机可用 | 暂不能多人协作 | 低 |

## 6. 推荐下一步

下一步不要继续扩写文档，而是改网页：

1. 优化关卡 2 表单，让它支持 3 个样本对象。
2. 把关卡 1 真实样本预置为演示数据导入能力。
3. 增加「复制阶段汇报卡」按钮。
4. 增加「报告预览」区域的信息层级。

这样项目会更像真正 PDE，而不是停留在策略材料。
