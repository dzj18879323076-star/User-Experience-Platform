# Working Guide：用户体验平台工作法

## 1. 使用原则

这个项目沉淀的是可复用工作法和产品能力，不迁移个人身份、内部工具、内部链接、权限信息、表名、域名、token、cookie 或历史备份材料。

默认处理顺序：

1. 先盘点本地目录、文件类型和已有文档。
2. 再判断当前任务属于产品方向、体验试跑、Web 实现、报告输出还是外部同步。
3. 对业务分析先讲 Why / So What，再讲 How。
4. 对工程实现先确认当前状态、改动范围和验证方式。
5. 对外部发送、上传、发布、删除、覆盖、批量移动等动作先确认。

## 2. 业务分析框架

围绕生活服务评价生产和看评消费，优先按以下顺序分析：

1. 对象：地点评价、商品评价、套餐评价、服务评价分别是什么。
2. 路径：用户在什么时机、从什么入口、用什么工具生产或消费评价。
3. 漏斗：触达、点击、开始、完成、有图/视频、有效、优质、分发。
4. 动机：自然表达、交易反馈、物质激励、荣誉激励、任务活动。
5. 质量：有效、优质、低质、虚假、激励、自然的口径。
6. 应用：用户端消费决策、商家端经营诊断、搜索推荐、内容分发、治理。
7. 方案：问题 -> 规模 -> 价值/ROI -> 方案 -> 风险/验证。

## 3. 文档输出结构

业务或分析结论默认使用：

```md
## 结论
## 背景/问题
## 关键发现
## 方案建议
## 风险与验证
## 待补信息
```

不是所有输出都要写成 PRD。给决策者看判断，给执行者看规则。

## 5. PDE 推进规则

- 当前项目优先做真实可用体验，不只写策略文档。
- 大改动先更新 PRD 或技术方案，再实现最小可用切片。
- Web MVP 当前是静态 HTML/CSS/JS，本地数据存在 `localStorage`。
- 先用静态站点验证交互；多人协作、后端存储、AI Game Master、飞书同步都属于后续迭代。
- 每完成一个可验证阶段，记录变更、验证结果和下一步。

## 6. 安全边界

- 不处理 cookie、token、私钥、密码等凭据。
- 不把公司内部材料上传到外部系统。
- 对敏感或未脱敏材料，只在本地做摘要和方法抽象。
- 对外发送、上传、发布前必须确认。
- 删除、覆盖、批量移动用户文件前必须确认。
